Phase 1 — Accounts, Keys & Software Prerequisites

This is the GATE phase. Every single item here must be completed before you do anything else. Missing any one of these will block you mid-installation.

1.1 Red Hat Account

  • Go to https://console.redhat.com
  • Sign in or create a free Red Hat account (use a corporate or personal email).
  • If you only have a developer/trial account: go to https://developers.redhat.com — sign up for the free Red Hat Developer Program.
  • This gives you access to OCP pull secrets and the Assisted Installer for non-production use.
  • After login, go to: OpenShift → Downloads → Pull Secret
  • Click Copy pull secret (or Download) — save this as pull-secret.json in a safe folder.
✅ TIP: The Pull Secret is a JSON blob that looks like: {"auths":{"cloud.openshift.com":{...}, "quay.io":{...}, ...}}. It contains credentials for Red Hat's container registries. Keep it secret — treat it like a password file.

1.2 Generate SSH Key Pair (on Bastion)

Run this on your bastion machine (Ubuntu terminal or WSL2):

ssh-keygen -t ed25519 -C "sno-admin" -f ~/.ssh/sno_key
# When prompted for passphrase: press Enter twice (no passphrase for lab use)

# Verify the keys were created:
ls -la ~/.ssh/sno_key*
# You should see:  sno_key  (private)  and  sno_key.pub  (public)

# Display the public key — you will paste this into the Assisted Installer:
cat ~/.ssh/sno_key.pub
⚠ NOTE: Copy the FULL contents of sno_key.pub — it starts with ssh-ed25519 AAAA... and ends with sno-admin. You will paste this exact string into the Assisted Installer.

Phase 2 — Bastion Host Full Setup

The bastion machine is your command center. Every installer command, DNS query, and OCP management action runs from here. Set it up completely before touching the OptiPlex.

2.1 Set a Static IP on the Bastion (Ubuntu)

Your bastion needs a permanent IP so DNS clients can always find it. Find your network interface name first:

ip addr show
# Look for your ethernet adapter — usually: eth0, enp2s0, ens3, etc.
# It will show your current IP next to 'inet'

Edit Netplan to set a static IP:

sudo nano /etc/netplan/00-installer-config.yaml

# Replace contents with (adjust interface name, IP, gateway to your network):
network:
  version: 2
  ethernets:
    enp2s0:                        # <-- replace with your actual interface name
      dhcp4: false
      addresses: [192.168.1.10/24] # <-- your bastion's static IP
      gateway4: 192.168.1.1        # <-- your router's IP
      nameservers:
        addresses: [127.0.0.1]     # <-- bastion points to itself for DNS (dnsmasq)

# Apply the configuration
sudo netplan apply

# Verify the IP is set
ip addr show enp2s0

2.2 Set a Static IP on the OptiPlex 5080 (Router DHCP Reservation)

The OptiPlex runs Windows now, but after SNO install it runs RHCOS. The easiest way to guarantee a static IP is a DHCP reservation in your router, pinned to the machine's MAC address.

  • On the OptiPlex 5080 (while it still runs Windows), open Command Prompt and run:
ipconfig /all
  • Look for the line: Physical Address: XX-XX-XX-XX-XX-XX — write this down.
  • Log into your router admin page (usually http://192.168.1.1 or http://192.168.0.1)
  • Find 'DHCP Reservation' or 'Static DHCP' or 'Address Reservation'
  • Add an entry: MAC = [MAC from step 1], IP = 192.168.1.50 (your chosen SNO IP)
  • Save and reboot router if required.
✅ TIP: Alternatively, you can set a static IP directly in the Assisted Installer's network configuration step — we will cover that in Phase 5.

2.3 Verify Network Connectivity

# From bastion — ping your router
ping -c 3 192.168.1.1

# Ping internet (should work)
ping -c 3 8.8.8.8

# Ping the OptiPlex (should work while Windows is running)
ping -c 3 192.168.1.50

Phase 3 — DNS Setup with dnsmasq

DNS is the most common failure point for OCP installations. OCP's Assisted Installer actually validates DNS before it lets you proceed. Get this right and the install will be smooth.

You are setting up dnsmasq on your bastion machine to act as the DNS server for your entire lab network. It will:

  • Resolve OCP cluster names (api.sno.mas.local, *.apps.sno.mas.local) to the SNO node IP
  • Forward all other DNS queries to the internet (Google 8.8.8.8)

3.1 Configure dnsmasq

Edit the main dnsmasq configuration file:

sudo mv /etc/dnsmasq.conf /etc/dnsmasq.conf.bak
sudo nano /etc/dnsmasq.conf

Paste the following (adjust IPs and domain to match your values):

# ─── dnsmasq.conf for SNO OCP Lab ───────────────────────────────────

# Listen on the bastion's LAN interface (replace IP with your bastion IP)
listen-address=192.168.1.10
listen-address=127.0.0.1
bind-interfaces

# Upstream DNS servers (internet forwarding)
server=8.8.8.8
server=8.8.4.4

# Don't forward queries without a domain to the internet
domain-needed

# Don't forward reverse lookups for private IPs to internet
bogus-priv

# Cache 1000 entries
cache-size=1000

# ─── OCP SNO DNS Records ─────────────────────────────────────────────
# Replace 192.168.1.50 with your actual SNO node IP
# Replace sno.mas.local with your actual cluster.domain

# OCP API endpoint
address=/api.sno.mas.local/192.168.1.50

# OCP Internal API (same IP for SNO)
address=/api-int.sno.mas.local/192.168.1.50

# Wildcard for ALL application routes (* .apps.sno.mas.local)
address=/.apps.sno.mas.local/192.168.1.50

# Explicit hostname for the SNO node itself (optional but recommended)
address=/sno-node.sno.mas.local/192.168.1.50

# ─────────────────────────────────────────────────────────────────────

3.2 Start and Enable dnsmasq

sudo systemctl start dnsmasq
sudo systemctl enable dnsmasq
sudo systemctl status dnsmasq
# Should show: Active: active (running)

3.3 Configure Firewall to Allow DNS

# If UFW is active (Ubuntu firewall):
sudo ufw allow from 192.168.1.0/24 to any port 53 proto udp
sudo ufw allow from 192.168.1.0/24 to any port 53 proto tcp
sudo ufw reload

# Check UFW status
sudo ufw status

3.4 Test DNS Resolution (CRITICAL)

These tests MUST all pass before you proceed:

Test 1: API endpoint resolves to SNO IP

dig @192.168.1.10 api.sno.mas.local
# Expected output: should show 192.168.1.50 in the ANSWER SECTION

Test 2: Internal API resolves to SNO IP

dig @192.168.1.10 api-int.sno.mas.local
# Expected: 192.168.1.50

Test 3: Wildcard apps route resolves to SNO IP

dig @192.168.1.10 anything.apps.sno.mas.local
dig @192.168.1.10 console-openshift-console.apps.sno.mas.local
# Expected: 192.168.1.50 for both

Test 4: Internet forwarding still works

dig @192.168.1.10 google.com
# Expected: Should resolve to Google's IPs (not 192.168.1.50)
🚨 CRITICAL: Do NOT continue to Phase 4 until ALL four DNS tests pass. If the Assisted Installer cannot resolve the DNS records, installation will fail at the network validation step.

3.5 Point Bastion and All Client PCs to dnsmasq

Every machine that needs to access the OCP console or Maximo Manage must use your bastion as its DNS server.

On the Bastion (already done in Phase 2)

The bastion's /etc/resolv.conf already has nameserver 127.0.0.1.

On Other Windows PCs / Laptops on the Same Network

  • Open Network & Internet Settings → Change adapter options
  • Right-click your network adapter → Properties → Internet Protocol Version 4 (IPv4)
  • Select 'Use the following DNS server addresses'
  • Preferred DNS server: 192.168.1.10 (your bastion IP)
  • Alternate DNS server: 8.8.8.8
  • Click OK → Close

Option: Configure Router to Push dnsmasq to All DHCP Clients

Alternatively, set your router's DNS server field to 192.168.1.10. This automatically gives every device on the network access to your DNS — no per-machine configuration needed.

Phase 4 — Network & Pre-Install Preparation

4.1 Verify Internet Access from Bastion

The bastion needs to reach these specific domains (used by the Assisted Installer and OCP install):

curl -I https://api.openshift.com
curl -I https://quay.io
curl -I https://registry.redhat.io
curl -I https://mirror.openshift.com
# All should return HTTP 200 or 301/302 (not connection refused or timeout)

4.2 Verify OptiPlex BIOS Settings

Boot the OptiPlex 5080 into BIOS (press F2 at startup) and verify:

The following table:

BIOS Setting Required Value How to Find It
Secure Boot DISABLED Security → Secure Boot → Disable
UEFI Boot Mode UEFI (not Legacy/CSM) General → Boot Sequence → UEFI
Virtualization (VT-x) ENABLED Virtualization Support → Enable VT for Direct I/O
Boot from USB ENABLED / first in boot order General → Boot Sequence — move USB to top
Network Boot (PXE) Not required — can be disabled Optional
🚨 CRITICAL: Secure Boot MUST be disabled. RHCOS (Red Hat CoreOS) installs fine on UEFI, but Secure Boot will block unsigned kernels and prevent the system from booting after installation.

4.3 Prepare USB Drive for Discovery ISO

You need a USB drive of at least 8 GB. The discovery ISO is typically 1-2 GB.

Option A: Write ISO on Linux (bastion)

# AFTER downloading the ISO from Assisted Installer (Phase 5):

# Identify your USB device (plug it in, then run:)
lsblk
# Look for the USB drive — e.g., /dev/sdb, /dev/sdc
# Verify the size matches your USB drive (e.g., 8GB = ~7.5G listed)

# UNMOUNT it first if auto-mounted
sudo umount /dev/sdX1   # replace X with your actual letter

# Write the ISO (REPLACE /dev/sdX with your actual USB device — e.g., /dev/sdb)
# WARNING: This will erase everything on the USB drive
sudo dd if=~/sno-install/discovery.iso of=/dev/sdX bs=4M status=progress oflag=sync
sync

Option B: Write ISO on Windows

  • Download Rufus from https://rufus.ie (free, no install needed)
  • Insert USB drive → Select the ISO file → Select 'DD Image' write mode → Click START
⚠ NOTE: In Rufus, you MUST select 'DD Image' mode (not ISO mode) when prompted. This is asked after you click Start.

Phase 5 — Generating the Discovery ISO (Assisted Installer)

The Red Hat Assisted Installer is a web-based wizard at console.redhat.com. It builds a customized bootable ISO specific to your cluster. When your OptiPlex boots from this ISO, it connects back to Red Hat's console and the installer completes the process.

5.1 Open the Assisted Installer

  • Open a browser on your bastion (or any PC with internet access)
  • Go to: https://console.redhat.com/openshift
  • Log in with your Red Hat account
  • Click the blue button: 'Create cluster'
  • Select the 'Datacenter' tab
  • Click: 'Create cluster' under 'Assisted Installer'

5.2 Cluster Details Page

Fill in the form exactly as follows:

The following table:

Field What to Enter Example
Cluster name Your chosen cluster name (NO spaces, lowercase) sno
Base domain Your chosen domain (NO http, just the domain) mas.local
OpenShift version Select latest stable 4.16.x or 4.17.x 4.16.45
CPU architecture x86_64 x86_64
High Availability Mode SINGLE NODE — select 'Install single node OpenShift (SNO)' SNO
⚠ NOTE: The Cluster name + Base domain combination is PERMANENT. Double check before clicking Next. If you made a mistake, delete the cluster and start a new one.

Click 'Next' after filling in all fields

5.3 Operators Page

You will see optional operators to install. For the MAS trial:

  • OpenShift Virtualization — NOT needed now
  • OpenShift Data Foundation — NOT needed (we will use LVM instead)
  • Leave all checkboxes empty
  • Click 'Next'

5.4 Host Discovery — ISO Configuration

5.4.1 Paste Your Pull Secret

  • You will see a field for 'Pull secret'
  • Open your pull-secret.json file on the bastion:
cat ~/sno-install/pull-secret.json
  • Copy the entire JSON content
  • Paste it into the Pull secret field in the browser

5.4.2 SSH Public Key

  • Open a terminal on the bastion and display your public key:
cat ~/.ssh/sno_key.pub
  • Copy the entire output (starts with ssh-ed25519 AAAA...)
  • Paste it into the 'SSH public key' field

5.4.3 Image Type

  • Select: Full image file — Includes everything needed for offline environments

  • Click 'Generate Discovery ISO'

⚠ NOTE: The ISO is valid for a limited time. Use it within a few days of generation.

5.5 Write the ISO to USB and Boot

Follow the steps in Phase 4.3 to write the ISO to your USB drive, then:

  • Plug the USB into the OptiPlex 5080
  • Power ON the OptiPlex
  • Press F12 when the Dell logo appears (boot menu)
  • Select the USB drive from the boot menu
  • The machine will boot into the RHCOS Live environment
  • It shows a terminal output screen — this is normal. Do not interact with it.
  • Back in your browser at console.redhat.com — the OptiPlex will appear in the 'Host Discovery' section within 3-5 minutes
⚠ NOTE: If the host doesn't appear after 5 minutes: check that your router assigned the correct IP to the OptiPlex, verify network cables are connected, and ensure the OptiPlex can reach the internet (needed for check-in to Red Hat's Assisted Installer service).

Phase 6 — Running the SNO Installation

Once the OptiPlex boots from the ISO and appears in the Assisted Installer, you complete the install wizard. The actual OCP install then runs unattended.

6.1 Host Discovery — Configure the Discovered Host

  • In the browser, you will see a row appear under 'Host inventory' for your discovered machine
  • It shows: hostname (auto-generated), status 'Discovering...' then 'Ready'
  • Once it shows 'Ready', click the host row to expand it
  • Verify the details:
  • CPU: should show 8 cores / 16 threads
  • Memory: should show ~31 GB available
  • Disk: should show the 477 GB NVMe SSD
  • Optionally rename the hostname by clicking the pencil icon — e.g., set to 'sno.mas.local'

6.1.1 Configure Static IP in the Installer

If you did NOT set up a DHCP reservation in your router, set the static IP here:

  • Click on the host row → 'Edit host' → 'Network configuration'
  • Set:
  • IP Address: 192.168.1.50 (your chosen SNO IP)
  • Prefix length: 24 (for /24 subnet)
  • Gateway: 192.168.1.1 (your router)
  • DNS: 192.168.1.10 (your bastion running dnsmasq)
  • Click Save

6.2 Storage Selection

  • Scroll down to 'Installation Disks'
  • Your 477 GB SSD should be listed — select it as the installation disk
  • If you see multiple disks — select the PRIMARY SSD (the largest)
⚠ NOTE: The selected disk will be COMPLETELY erased and formatted as RHCOS. Everything on it is gone.

6.3 Networking — Cluster Networking

  • Click 'Next' to proceed to the Networking page
  • The installer shows detected network interfaces and subnets
  • Under 'Cluster network CIDR': leave defaults (10.128.0.0/14)
  • Under 'Service network CIDR': leave defaults (172.30.0.0/16)
  • Machine network should show: 192.168.1.0/24 (your LAN subnet)
  • Verify that the API VIP and Ingress VIP fields are NOT shown (SNO doesn't need them — it uses the node IP directly)
✅ TIP: In SNO mode, there is no VIP (Virtual IP). Both API and app traffic go directly to the single node's IP. This is simpler but means there's no failover.

6.4 Review & Validation

  • Click 'Next' to reach the Review page
  • The installer runs automatic validations. You should see green checkmarks for:
  • DNS Wildcard: *.apps.sno.mas.local → 192.168.1.50 ✓
  • API DNS: api.sno.mas.local → 192.168.1.50 ✓
  • API-Int DNS: api-int.sno.mas.local → 192.168.1.50 ✓
  • Network connectivity ✓
  • NTP (time sync) ✓
  • Disk available ✓
🚨 CRITICAL: If any DNS validation FAILS (shows red): do NOT proceed. Go back to Phase 3, re-verify your dnsmasq config, restart dnsmasq, and re-run the DNS dig tests. The install will fail if DNS is broken.

6.5 Start the Installation

Once ALL validations show green, click 'Install cluster' A progress bar appears — track the phases:

The following table:

Phase What's Happening Approx Time
Preparing Generating ignition configs, writing to ISO bootstrap 2-5 min
Installing Writing RHCOS to disk, applying ignition config 10-20 min
Rebooting Machine reboots from disk (remove USB after first reboot) 5 min
Configuring control plane etcd, API server, controller manager starting 15-25 min
Bootstrapping Cluster operators deploying (40+ operators) 20-40 min
Finalizing Last operators reconciling, ingress deploying 5-10 min
⚠ NOTE: Total installation time: 45-90 minutes. The machine reboots 2-3 times during the process. This is normal. Do NOT power off the machine.

6.6 Monitor Installation Progress

Watch the Assisted Installer progress bar in your browser. You can also ssh into the node during installation (using the key you provided) to watch logs:

# From bastion — SSH into the SNO node during install (RHCOS default user is 'core')
ssh -i ~/.ssh/sno_key core@192.168.1.50

# Watch bootstrap logs during install
sudo journalctl -b -f

# Or watch specific cluster operator status
sudo journalctl -u release-image.service -f

6.7 Remove the USB Drive

After the installation writes the OS to disk and the machine reboots for the first time (~15-20 minutes in), remove the USB drive. If the machine boots back into the USB, it will restart the installation from scratch.

✅ TIP: Watch the Assisted Installer — when it shows 'Writing disk image complete' and the machine reboots, immediately remove the USB. The machine must then boot from the internal SSD.

6.8 Installation Complete

When the Assisted Installer shows 'Installation complete' and 'Cluster installed successfully':

  • The OCP web console URL is displayed: https://console-openshift-console.apps.sno.mas.local
  • Click 'Download kubeconfig' — save as: ~/sno-install/kubeconfig
  • Note the kubeadmin password shown on screen — copy it somewhere safe
  • Click 'Open Web Console' — your browser opens the OCP console
⚠ NOTE: If the browser shows a certificate warning (SSL), this is expected — SNO generates self-signed certificates by default. Click 'Advanced' → 'Accept the risk and continue' in Firefox, or 'Proceed anyway' in Chrome.

Phase 7 — Post-Install OCP Verification

Installation complete does not mean the cluster is fully healthy. You must verify that all cluster operators and nodes are in a good state before proceeding to storage or MAS.

7.1 Configure kubectl/oc Access on Bastion

# Copy the kubeconfig you downloaded
mkdir -p ~/.kube
cp ~/sno-install/kubeconfig ~/.kube/config

# Set permissions
chmod 600 ~/.kube/config

# Verify connection to the cluster
oc whoami
# Expected output: system:admin

oc cluster-info
# Expected: shows API server URL and console URL

7.2 Verify Node Status

oc get nodes
# Expected output:
# NAME                     STATUS   ROLES                         AGE   VERSION
# sno.mas.local            Ready    control-plane,master,worker   15m   v1.29.x

# If still NotReady after 20 min — check node logs (Phase 7.5)

7.3 Verify Cluster Operators

This is the most important check. All cluster operators must be AVAILABLE=True and not PROGRESSING or DEGRADED.

oc get clusteroperators
# You should see ~30+ operators listed

Phase 8 — LVM Storage Operator Setup

OpenShift requires Persistent Volumes (PVs) for many workloads — including the internal image registry and later for MAS/Manage databases. On bare metal SNO with a single SSD, the LVM Storage Operator creates a storage class that carves PVs from available disk space.

8.1 Find Available Disks on the Node

First, check what disk partitions exist and what space is available:

# SSH into the node
ssh -i ~/.ssh/sno_key core@192.168.1.50

# List all block devices
lsblk
# You should see:
#  nvme0n1  (main SSD — this is the boot disk)
#    nvme0n1p1  (EFI partition)
#    nvme0n1p2  (RHCOS boot)
#    nvme0n1p3  (RHCOS root)
#    nvme0n1p4  (RHCOS var — this has most free space)

# Check free disk space
df -h /var

exit
⚠ NOTE: Since you only have ONE SSD (no second disk), the LVM Operator will create a thin-provisioned volume group on top of the available free space on your NVMe. This works for a trial but is less performant than a dedicated second disk.

8.2 Install the LVM Storage Operator via OCP Console

Step 1: Go to OperatorHub

  • In the OCP Web Console, click: Operators (left sidebar) → OperatorHub
  • In the search box, type: LVM Storage
  • You should see 'LVM Storage' from Red Hat — click it
  • Click 'Install'

Step 2: Configure Installation

The following table:

Field Value to Set
Update channel stable-4.16 (match your OCP version)
Installation mode A specific namespace on the cluster
Installed Namespace openshift-storage (create if prompted)
Update approval Automatic
  • Click 'Install'
  • Wait 2-3 minutes for the operator to install
  • Navigate to: Operators → Installed Operators → verify 'LVM Storage' shows 'Succeeded'

8.3 Create an LVMCluster Custom Resource

This step tells the LVM Operator to create the actual storage volume group on your disk.

Via the Web Console

  • In the OCP Console: Operators → Installed Operators → LVM Storage
  • Click 'Create instance' under LVMCluster
  • Switch to YAML view and paste the following:
apiVersion: lvm.topolvm.io/v1alpha1
kind: LVMCluster
metadata:
  name: my-lvmcluster
  namespace: openshift-storage
spec:
  storage:
    deviceClasses:
    - name: vg1
      thinPoolConfig:
        name: thin-pool-1
        sizePercent: 90
        overprovisionRatio: 10
  • Click 'Create'
  • Wait 2-3 minutes for the LVMCluster to show status 'Ready'

8.4 Alternatively: Install via CLI

If you prefer using the command line from the bastion:

Phase 9 — Image Registry Configuration

On bare metal OCP, the internal image registry is disabled after installation because there is no default storage class pre-configured. Now that LVM storage is ready, you must enable and configure it. This is required for many OCP features and is a mandatory prerequisite for MAS.

9.1 Enable the Image Registry

First, change the registry's managementState from Removed to Managed:

oc patch configs.imageregistry.operator.openshift.io cluster \
--type merge \
--patch '{"spec":{"managementState":"Managed"}}'

# Verify the patch was applied
oc get configs.imageregistry.operator.openshift.io cluster -o yaml | grep managementState
# Should show: managementState: Managed

9.2 Configure Persistent Storage for the Image Registry

Now attach the LVM storage to the registry using a PVC:

oc patch configs.imageregistry.operator.openshift.io cluster \
--type merge \
--patch '{
  "spec": {
    "storage": {
      "pvc": {
        "claim": ""
      }
    }
  }
}'

# This causes OCP to auto-create a PVC called 'image-registry-storage'
# Verify the PVC was created and is Bound:
oc get pvc -n openshift-image-registry
# Should show: image-registry-storage   Bound

9.3 Enable External Route to the Registry (Optional)

oc patch configs.imageregistry.operator.openshift.io cluster \
--type merge \
--patch '{"spec":{"defaultRoute":true}}'

# This creates a route: default-route-openshift-image-registry.apps.sno.mas.local
# Verify:
oc get route -n openshift-image-registry

9.4 Verify Registry Is Running

# Check registry pods
oc get pods -n openshift-image-registry
# Should show: image-registry-XXXX pod with STATUS = Running

# Check the cluster operator status
oc get co image-registry
# Should show: AVAILABLE=True, PROGRESSING=False, DEGRADED=False

# Verify the registry PVC is bound
oc get pvc -n openshift-image-registry
# STATUS should be: Bound

Cause How to Diagnose Fix

Error Message Cause Fix
