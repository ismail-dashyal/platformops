MAS Lab Deployment — Corrected Master Plan

Hardware Allocation Strategy

Node 1: The SNO Engine (Ryzen 9 / 256GB RAM)

  • 500GB SSD: Dedicated to Red Hat Enterprise Linux CoreOS (RHCOS). Holds the OpenShift control plane, etcd state, and container image registry.
  • 1TB SSD: Left raw during installation, later claimed by the Local Storage Operator (LSO) for high-speed RWO storage (Db2, Kafka/UDS).
  • RTX 5070: Optional passthrough via the NVIDIA GPU Operator for MAS Visual Inspection. Note: the GPU Operator requires the Node Feature Discovery (NFD) Operator as a prerequisite — install NFD first, or the GPU Operator install will fail to detect hardware.
  • Workload: OpenShift Container Platform, MAS Operator, Suite License Service (SLS), Db2 container, MongoDB (internal operator, recommended for SNO — see note below), and Maximo Manage application pods.

Sizing note: 256GB RAM is generous, but a full MAS stack (Core, Manage, Db2, Kafka/UDS, Cert-Manager, MongoDB) on a single-node cluster is resource-tight in practice. SNO is officially supported for MAS but has known limits on pod density and etcd performance under heavy churn. Monitor oc adm top nodes and etcd health during install; a compact (3-node) cluster is the recommended baseline if you hit instability.

Node 2: The Infrastructure Bastion (i7 / 32GB RAM / 477GB Storage)

  • OS: Ubuntu Server 22.04 LTS or RHEL 9.
  • The Traffic Cop: Runs dnsmasq and HAProxy to route all .apps.sno.mas.lab traffic into the SNO.
  • Shared Storage: NFS server providing the RWX storage class for Maximo attachments and logs.
  • Optional services (not mandatory for MAS, but supported choices): external MongoDB 7.0 replica set and OpenLDAP, if you prefer to externalize them rather than use MAS's bundled operators. See clarification below.

MongoDB clarification: MAS 9.x installs and manages its own internal MongoDB operator by default, which is the standard path for SNO. External MongoDB is only needed for specific HA/multi-cluster topologies or if you explicitly opt out of the bundled instance during Suite CR configuration. Treat external MongoDB as an architectural option, not a hard requirement.

OpenLDAP clarification: LDAP integration is optional. MAS Suite UI can use OpenShift's built-in OAuth (htpasswd, etc.) or App Connect-based identity providers for a lab environment. Only deploy OpenLDAP if you specifically want enterprise-style directory-based user management.

The RLKS (Rational License Key Server) runs inside a dedicated VM on the i7 bastion to preserve the existing host ID / MAC address binding tied to your IBM license, avoiding a rehost.

The i7 hypervisor must use Bridged Networking (not NAT) so the RLKS VM sits directly on the local subnet with its own static IP and licensed MAC address.

Phase 1: Network & Virtualization Pre-Flight

Component Address
Subnet 192.168.0.0/24
Gateway / Router 192.168.0.1
Ryzen SNO Node 192.168.0.20
i7 Bastion Host OS 192.168.0.10
RLKS Virtual Machine 192.168.0.11
Base Domain sno.mas.lab

Critical: In the i7 hypervisor (KVM/QEMU, VirtualBox, or VMware), edit the RLKS VM's network adapter to Bridged mode and manually set the MAC address to match the one your license.dat is tied to.

Phase 2: Building the i7 Infrastructure Bastion

1. Hypervisor & RLKS VM

  • Install hypervisor packages (qemu-kvm, libvirt-daemon-system, virt-manager).
  • Launch/import the existing RLKS VM. Confirm it binds to the bridged interface, takes static IP 192.168.0.11, and can ping both the host (192.168.0.10) and the router.
  • Confirm RLKS is listening on port 27000 (lmgrd). Port 27001 (ibmratl) is internal vendor-daemon traffic on the same host and generally doesn't need to be exposed externally — open it only if your specific license configuration requires direct vendor-daemon access. Open required ports on the VM's internal firewall.

2. Host OS Networking Services

  • DNS (dnsmasq) on 192.168.0.10: serve api.sno.mas.lab, api-int.sno.mas.lab, and wildcard *.apps.sno.mas.lab, all pointing to 192.168.0.10. Open port 53 (UDP/TCP). Double-check the api-int record matches the exact node hostname set during ISO generation — mismatches here are a common cause of bootstrap hangs.
  • HAProxy: frontend on ports 80, 443, 6443, 22623, all backending to the SNO at 192.168.0.20. Open these ports on the host firewall.

3. Host OS Containerized Core Services (as needed)

  • NFS Shared Storage Vault (required): nfs-kernel-server, export path /var/nfs/mas_rwx, entry /var/nfs/mas_rwx *(rw,sync,no_subtree_check,no_root_squash) in /etc/exports, then exportfs -ra. Open port 2049.
  • External MongoDB 7.0 (optional, only if not using MAS's internal Mongo operator): deploy via Docker/Podman with --replSet rs0, then run rs.initiate() in mongosh. Open port 27017.
  • OpenLDAP (optional): deploy via container (e.g., Bitnami), pre-populate an OU for Maximo admins/users. Open ports 389 and 636.

Phase 3: Ryzen SNO Node Installation

  • Red Hat Assisted Installer: generate a Single Node OpenShift Discovery ISO via console.redhat.com. Cluster name sno, base domain mas.lab. Set DNS to 192.168.0.10.
  • Targeted Disk Selection: select only the 500GB SSD as the install target. Leave the 1TB SSD raw and unformatted.
  • Trigger Installation: write RHCOS, bootstrap the cluster, remove the USB on restart.

Phase 4: IBM Entitlement Key & Pull Secret (previously missing)

Before installing any IBM operators (SLS, UDS, Db2, MAS), configure image pull access:

  • Generate an IBM Entitlement Key from container.softwarewarranty / MyIBM.
  • Create/update the global pull secret (or a namespace-scoped ibm-entitlement-key secret) with cp.icr.io credentials using the entitlement key as the password.
  • Verify operator catalog sources for IBM Operator Catalog and Cert-Manager are configured in OperatorHub.

Without this step, IBM operator images will fail to pull and all later phases will stall.

Phase 5: Cluster Storage Allocation

  • Local Storage Class (RWO): install the Local Storage Operator (LSO) from OperatorHub. Create a LocalVolume targeting the raw 1TB NVMe drive, producing storage class local-sc for Db2 and Kafka/UDS.
  • NFS Storage Class (RWX): deploy nfs-subdir-external-provisioner via Helm, pointing to 192.168.0.10 and /var/nfs/mas_rwx. Name it nfs-client.

Note: nfs-subdir-external-provisioner is community-maintained and not a Red Hat-supported CSI driver. Fine for a lab, but if you later need a support-matched configuration, use Red Hat's supported NFS CSI operator instead.

Phase 6: IBM Foundational Services Deployment

  • Cert-Manager: install the Red Hat Cert-Manager operator for internal certificates.
  • Suite License Service (SLS): install the IBM SLS operator. Configure it to connect to the RLKS VM via 27000@192.168.0.11 rather than uploading a license file.
  • User Data Services (UDS): deploy the UDS operator, pointing Kafka stream storage to local-sc.
  • IBM Db2 Warehouse: install the Db2 operator. Accept the Db2 license (LICENSE=accept parameter is required during provisioning). Provision the Maximo Manage database instance with storage bound to local-sc, sized at minimum ~100GB for a representative workload.

Phase 7: Maximo Application Suite Setup

  • MAS Operator: install from the OpenShift catalog.
  • Suite CR Initialization:
    • If using internal MongoDB (recommended for SNO), allow the MAS operator to provision it automatically — no external database block needed.
    • If using external MongoDB, point the database block to 192.168.0.10:27017.
    • If using OpenLDAP, configure the user management synchronization block to 192.168.0.10:389. Otherwise, configure OAuth-based identity via OpenShift.
  • Maximo Manage Activation: open the MAS core dashboard, select "Manage", bind the EAM application to the Db2 instance from Phase 6, and assign persistent storage to nfs-client for attachments, logs, and reports.

Summary of Corrections from Original Plan

  1. Added Phase 4 (Entitlement Key / pull secret) — previously missing entirely and would have blocked all IBM operator installs.
  2. Reframed external MongoDB as optional — MAS 9.x defaults to an internal MongoDB operator on SNO.
  3. Reframed OpenLDAP as optional — OpenShift OAuth is a valid alternative for a lab.
  4. Added Db2 license acceptance (LICENSE=accept) and minimum storage sizing guidance.
  5. Added NFD Operator as a prerequisite for the GPU Operator.
  6. Clarified RLKS port usage (27000 vs 27001).
  7. Added a DNS troubleshooting note re: api-int hostname matching.
  8. Flagged nfs-subdir-external-provisioner as unsupported (community) for production-parity environments.
  9. Added SNO resource-density caution for the full MAS+Db2+UDS stack.
