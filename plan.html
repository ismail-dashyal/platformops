MAS Lab Deployment — Corrected Master Plan[cite: 4]

Hardware Allocation Strategy[cite: 4]

Node 1: The SNO Engine (Ryzen 9 / 256GB RAM)[cite: 4]

  • 500GB SSD: Dedicated to Red Hat Enterprise Linux CoreOS (RHCOS).[cite: 4] Holds the OpenShift control plane, etcd state, and container image registry.[cite: 4]
  • 1TB SSD: Left raw during installation, later claimed by the Local Storage Operator (LSO) for high-speed RWO storage (Db2, Kafka/UDS).[cite: 4]
  • RTX 5070: Optional passthrough via the NVIDIA GPU Operator for MAS Visual Inspection.[cite: 4] Note: the GPU Operator requires the Node Feature Discovery (NFD) Operator as a prerequisite — install NFD first, or the GPU Operator install will fail to detect hardware.[cite: 4]
  • Workload: OpenShift Container Platform, MAS Operator, Suite License Service (SLS), Db2 container, MongoDB (internal operator, recommended for SNO — see note below), and Maximo Manage application pods.[cite: 4]

Sizing note: 256GB RAM is generous, but a full MAS stack (Core, Manage, Db2, Kafka/UDS, Cert-Manager, MongoDB) on a single-node cluster is resource-tight in practice.[cite: 4] SNO is officially supported for MAS but has known limits on pod density and etcd performance under heavy churn.[cite: 4] Monitor oc adm top nodes and etcd health during install; a compact (3-node) cluster is the recommended baseline if you hit instability.[cite: 4]

Node 2: The Infrastructure Bastion (i7 / 32GB RAM / 477GB Storage)[cite: 4]

  • OS: Ubuntu Server 22.04 LTS or RHEL 9.[cite: 4]
  • The Traffic Cop: Runs dnsmasq and HAProxy to route all .apps.sno.mas.lab traffic into the SNO.[cite: 4]
  • Shared Storage: NFS server providing the RWX storage class for Maximo attachments and logs.[cite: 4]
  • Optional services (not mandatory for MAS, but supported choices): external MongoDB 7.0 replica set and OpenLDAP, if you prefer to externalize them rather than use MAS's bundled operators.[cite: 4] See clarification below.[cite: 4]

MongoDB clarification: MAS 9.x installs and manages its own internal MongoDB operator by default, which is the standard path for SNO.[cite: 4] External MongoDB is only needed for specific HA/multi-cluster topologies or if you explicitly opt out of the bundled instance during Suite CR configuration.[cite: 4] Treat external MongoDB as an architectural option, not a hard requirement.[cite: 4]

OpenLDAP clarification: LDAP integration is optional.[cite: 4] MAS Suite UI can use OpenShift's built-in OAuth (htpasswd, etc.) or App Connect-based identity providers for a lab environment.[cite: 4] Only deploy OpenLDAP if you specifically want enterprise-style directory-based user management.[cite: 4]

The RLKS (Rational License Key Server) runs inside a dedicated VM on the i7 bastion to preserve the existing host ID / MAC address binding tied to your IBM license, avoiding a rehost.[cite: 4]

The i7 hypervisor must use Bridged Networking (not NAT) so the RLKS VM sits directly on the local subnet with its own static IP and licensed MAC address.[cite: 4]

Subnet[cite: 4] 192.168.0.0/24[cite: 4]
Gateway / Router[cite: 4] 192.168.0.1[cite: 4]
Ryzen SNO Node[cite: 4] 192.168.0.20[cite: 4]
i7 Bastion Host OS[cite: 4] 192.168.0.10[cite: 4]
RLKS Virtual Machine[cite: 4] 192.168.0.11[cite: 4]
Base Domain[cite: 4] sno.mas.lab[cite: 4]

Critical: In the i7 hypervisor (KVM/QEMU, VirtualBox, or VMware), edit the RLKS VM's network adapter to Bridged mode and manually set the MAC address to match the one your license.dat is tied to.[cite: 4]

Phase 2: Building the i7 Infrastructure Bastion[cite: 4]

1. Hypervisor & RLKS VM[cite: 4]

  • Install hypervisor packages (qemu-kvm, libvirt-daemon-system, virt-manager).[cite: 4]
  • Launch/import the existing RLKS VM.[cite: 4] Confirm it binds to the bridged interface, takes static IP 192.168.0.11, and can ping both the host (192.168.0.10) and the router.[cite: 4]
  • Confirm RLKS is listening on port 27000 (lmgrd).[cite: 4] Port 27001 (ibmratl) is internal vendor-daemon traffic on the same host and generally doesn't need to be exposed externally — open it only if your specific license configuration requires direct vendor-daemon access.[cite: 4] Open required ports on the VM's internal firewall.[cite: 4]

2. Host OS Networking Services[cite: 4]

  • DNS (dnsmasq) on 192.168.0.10: serve api.sno.mas.lab, api-int.sno.mas.lab, and wildcard *.apps.sno.mas.lab, all pointing to 192.168.0.10.[cite: 4] Open port 53 (UDP/TCP).[cite: 4] Double-check the api-int record matches the exact node hostname set during ISO generation — mismatches here are a common cause of bootstrap hangs.[cite: 4]
  • HAProxy: frontend on ports 80, 443, 6443, 22623, all backending to the SNO at 192.168.0.20.[cite: 4] Open these ports on the host firewall.[cite: 4]

3. Host OS Containerized Core Services (as needed)[cite: 4]

  • NFS Shared Storage Vault (required): nfs-kernel-server, export path /var/nfs/mas_rwx, entry /var/nfs/mas_rwx *(rw,sync,no_subtree_check,no_root_squash) in /etc/exports, then exportfs -ra.[cite: 4] Open port 2049.[cite: 4]
  • External MongoDB 7.0 (optional, only if not using MAS's internal Mongo operator): deploy via Docker/Podman with --replSet rs0, then run rs.initiate() in mongosh.[cite: 4] Open port 27017.[cite: 4]
  • OpenLDAP (optional): deploy via container (e.g., Bitnami), pre-populate an OU for Maximo admins/users.[cite: 4] Open ports 389 and 636.[cite: 4]

Phase 3: Ryzen SNO Node Installation[cite: 4]

  • Red Hat Assisted Installer: generate a Single Node OpenShift Discovery ISO via console.redhat.com.[cite: 4] Cluster name sno, base domain mas.lab.[cite: 4] Set DNS to 192.168.0.10.[cite: 4]
  • Targeted Disk Selection: select only the 500GB SSD as the install target.[cite: 4] Leave the 1TB SSD raw and unformatted.[cite: 4]
  • Trigger Installation: write RHCOS, bootstrap the cluster, remove the USB on restart.[cite: 4]

Phase 4: IBM Entitlement Key & Pull Secret (previously missing)[cite: 4]

Before installing any IBM operators (SLS, UDS, Db2, MAS), configure image pull access:[cite: 4]

  • Generate an IBM Entitlement Key from container.softwarewarranty / MyIBM.[cite: 4]
  • Create/update the global pull secret (or a namespace-scoped ibm-entitlement-key secret) with cp.icr.io credentials using the entitlement key as the password.[cite: 4]
  • Verify operator catalog sources for IBM Operator Catalog and Cert-Manager are configured in OperatorHub.[cite: 4]

Without this step, IBM operator images will fail to pull and all later phases will stall.[cite: 4]

Phase 5: Cluster Storage Allocation[cite: 4]

  • Local Storage Class (RWO): install the Local Storage Operator (LSO) from OperatorHub.[cite: 4] Create a LocalVolume targeting the raw 1TB NVMe drive, producing storage class local-sc for Db2 and Kafka/UDS.[cite: 4]
  • NFS Storage Class (RWX): deploy nfs-subdir-external-provisioner via Helm, pointing to 192.168.0.10 and /var/nfs/mas_rwx.[cite: 4] Name it nfs-client.[cite: 4]

Note: nfs-subdir-external-provisioner is community-maintained and not a Red Hat-supported CSI driver.[cite: 4] Fine for a lab, but if you later need a support-matched configuration, use Red Hat's supported NFS CSI operator instead.[cite: 4]

Phase 6: IBM Foundational Services Deployment[cite: 4]

  • Cert-Manager: install the Red Hat Cert-Manager operator for internal certificates.[cite: 4]
  • Suite License Service (SLS): install the IBM SLS operator.[cite: 4] Configure it to connect to the RLKS VM via 27000@192.168.0.11 rather than uploading a license file.[cite: 4]
  • User Data Services (UDS): deploy the UDS operator, pointing Kafka stream storage to local-sc.[cite: 4]
  • IBM Db2 Warehouse: install the Db2 operator.[cite: 4] Accept the Db2 license (LICENSE=accept parameter is required during provisioning).[cite: 4] Provision the Maximo Manage database instance with storage bound to local-sc, sized at minimum ~100GB for a representative workload.[cite: 4]

Phase 7: Maximo Application Suite Setup[cite: 4]

  • MAS Operator: install from the OpenShift catalog.[cite: 4]
  • Suite CR Initialization:[cite: 4]
    • If using internal MongoDB (recommended for SNO), allow the MAS operator to provision it automatically — no external database block needed.[cite: 4]
    • If using external MongoDB, point the database block to 192.168.0.10:27017.[cite: 4]
    • If using OpenLDAP, configure the user management synchronization block to 192.168.0.10:389.[cite: 4] Otherwise, configure OAuth-based identity via OpenShift.[cite: 4]
  • Maximo Manage Activation: open the MAS core dashboard, select "Manage", bind the EAM application to the Db2 instance from Phase 6, and assign persistent storage to nfs-client for attachments, logs, and reports.[cite: 4]

  3. Reframed OpenLDAP as optional — OpenShift OAuth is a valid alternative for a lab.[cite: 4]
  4. Added Db2 license acceptance (LICENSE=accept) and minimum storage sizing guidance.[cite: 4]
  5. Added NFD Operator as a prerequisite for the GPU Operator.[cite: 4]
  6. Clarified RLKS port usage (27000 vs 27001).[cite: 4]
